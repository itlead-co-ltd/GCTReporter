分析以下需求，提出结构化的澄清问题清单，确保需求**完整、可执行、符合{domain}行业规范**。

**输出要求**：

1. **完整性**: 覆盖功能、非功能、合规、技术、管理5大维度

3. **专业性**: 引用行业标准、法规条款、竞品数据
4. **优先级**: 区分P0（必须）、P1（应该）、P2（可以）三级

**选项设计6大原则**:

1. **互斥性**: 选项之间不重叠

3. **可选性**: 避免"是/否"二选一，提供3-5个选项
4. **具体性**: 每个选项有明确的场景描述
5. **推荐性**: 标记推荐选项（基于行业最佳实践）
6. **可扩展性**: 允许用户补充"其他"选项

### 8. 优先级排序建议

**说明**: 按重要性和紧急性对所有问题进行分级

**优先级判断决策树**:

```
是否直接影响项目可行性？
├── 是 → P0（立即澄清）
│   ├── 不澄清会导致项目无法启动
│   ├── 不澄清会导致重大返工（成本>30%预算）
│   └── 涉及法律红线或强制合规要求
└── 否 → 继续判断
    是否影响核心功能设计？
    ├── 是 → P1（尽快澄清）
    │   ├── 影响数据模型设计
    │   ├── 影响技术架构选型
    │   └── 影响用户核心体验
    └── 否 → P2（可后续澄清）
        ├── 属于优化类需求
        ├── 属于扩展类功能
        └── 可以在迭代中逐步明确
```

### 🗺️ 建议处理路径

**根据需求层次选择路径**:

#### 路径A: Epic级需求（建议采用"3步走"策略）

**第一步：战略对齐**（1-2周）

- 目标: 澄清P0级别问题（预算、时间、核心业务模式）
- 交付物:
  - [ ] 项目章程（Project Charter）
  - [ ] 干系人地图（Stakeholder Map）
  - [ ] 技术选型大方向确认（云原生 vs 传统架构）
- 里程碑: 高层决策会通过

**第二步：MVP设计**（2-4周）

- 目标: 聚焦核心用户场景，拆分为可独立交付的Feature
- 交付物:
  - [ ] 用户故事地图（User Story Map）
  - [ ] MVP功能清单（含优先级）
  - [ ] 交互原型（低保真/高保真）
- 里程碑: 原型评审通过

**第三步：迭代优化**（3-6个月）

- 目标: 基于MVP反馈逐步完善功能，补充P1和P2需求
- 交付物:
  - [ ] 迭代计划（Sprint Plan）
  - [ ] 运营数据报表
  - [ ] 优化需求清单
- 里程碑: 正式上线

---

## 📚 附录：关键术语解释

| 术语               | 解释                                                                              | 示例                                                 |
| ------------------ | --------------------------------------------------------------------------------- | ---------------------------------------------------- |
| **EPIC**     | 敏捷开发中的史诗级需求，通常需要拆分为多个Feature才能实施，周期跨度数月           | "构建XXX会议论坛全年线上线下活动管理平台"            |
| **FEATURE**  | 可独立交付价值的功能模块，通常1-2个月完成                                         | "用户注册认证模块""活动报名支付模块"                 |
| **STORY**    | 单一用户故事，1-2周可完成                                                         | "作为参展企业，我要报名参加论坛，以便获取入场凭证"   |
| **MVP**      | Minimum Viable Product（最小可行产品），用最小成本快速验证核心假设                | 先实现"发布活动-报名-签到"闭环，暂不做营销推广功能   |
| **MECE**     | Mutually Exclusive, Collectively Exhaustive（互斥且完整），麦肯锡问题分析法       | 支付方式选项：A.微信 B.支付宝 C.银联 D.PayPal E.其他 |
| **P0/P1/P2** | 优先级分级，P0=Must Have（必须），P1=Should Have（应该），P2=Nice to Have（可以） | P0：预算确认，P1：技术选型，P2：多语言支持           |
| **SLA**      | Service Level Agreement（服务等级协议），明确系统可用性指标                       | 可用性99.9%，响应时间≤2秒                           |
| **QPS**      | Queries Per Second（每秒查询率），衡量系统并发能力                                | 日常100 QPS，高峰1000 QPS                            |
| **MTTR**     | Mean Time To Repair（平均修复时间），衡量运维响应速度                             | MTTR≤30分钟                                         |
| **NPS**      | Net Promoter Score（净推荐值），衡量用户满意度                                    | NPS≥50分为优秀                                      |
| **OMO**      | Online-Merge-Offline（线上线下融合），新零售/会展行业趋势                         | 线上报名、线下签到核销                               |
| **GDPR**     | General Data Protection Regulation（欧盟通用数据保护条例）                        | 处理境外用户数据需遵守                               |
| **等保三级** | 国家信息安全等级保护三级认证，针对重要信息系统的强制要求                          | 金融/政务系统必须通过                                |
| **CA证书**   | Certificate Authority（数字证书认证机构），用于电子签名                           | 电子合同需要CA证书                                   |

## 📧 使用说明

### 如何使用本模板

1. **填充领域信息**:

   - 将 `{domain}` 替换为具体行业（如"对外贸易/会展经济"）
   - 将 `{compliance_requirements}` 替换为关键法规（如"《数据出境安全评估办法》"）
   - 将 `{domain_constraints}` 替换为行业约束（如"必须内网部署"）
2. **粘贴原始需求**:

   - 将客户提供的需求描述粘贴到"输入内容"部分

   - 使用AI大模型（如DeepSeek/GPT-4）基于模板生成问题清单
   - 人工审核并补充行业专业内容
4. **组织澄清会议**:

   - 使用"推荐会议形式"章节的议程模板
   - 会前发送问题清单给客户预习
5. **持续迭代**:

   - 根据澄清结果更新需求文档
   - 将新的问题和答案沉淀到知识库

## 🎓 最佳实践建议

### 产品经理使用技巧

1. **前置准备**:

   - 在需求澄清会前，先做竞品分析和行业调研
   - 准备2-3个竞品系统的演示视频或截图
   - 准备技术方案的对比表（Excel）
2. **提问技巧**:

   - 先问开放式问题了解背景（"您为什么需要这个功能？"）
   - 再用本模板的封闭式问题收敛答案（"您选择方案A还是方案B？"）
   - 对于模糊回答，用"假设法"引导（"假设预算100万和200万，功能有什么区别？"）
3. **会议控制**:

   - 严格控制每个问题讨论时间（5分钟）
   - 对于无法当场决策的问题，记录为"待确认"，会后跟进
   - 会议结束前必须明确"下次会议时间"和"准备事项"
4. **文档管理**:

   - 每次会议后24小时内发送会议纪要
   - 使用版本控制工具管理需求文档（Git/SVN）
   - 建立需求变更日志（谁/何时/改了什么/为什么）

### 常见误区

❌ **错误做法1**: 一次会议试图澄清所有问题
✅ **正确做法**: 分2-3次会议，每次聚焦不同主题（业务需求/技术方案/原型确认）

❌ **错误做法2**: 只提开放式问题（"您想要什么功能？"）
✅ **正确做法**: 提供可选答案，降低客户决策成本（"您选A还是B？"）

❌ **错误做法3**: 所有问题都是P0优先级
✅ **正确做法**: 严格控制P0数量（≤6个），真正的"Must Have"

❌ **错误做法4**: 只澄清功能需求，忽略非功能需求
✅ **正确做法**: 非功能需求往往是项目失败的主因，必须明确量化指标

❌ **错误做法5**: 澄清完就开始开发，不做原型验证
✅ **正确做法**: Epic/Feature级需求必须先做原型，快速验证理解是否一致
