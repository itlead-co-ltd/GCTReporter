# 项目状态文档编写规范

> **文档目的**: 为项目状态跟踪文档的编写和维护提供标准化规范  
> **适用范围**: 软件开发项目的状态跟踪和进度管理  
> **版本**: v1.0  
> **制定日期**: 2025-09-30

---

## 📋 文档结构规范

### 1. 必需的一级章节 (按顺序)

```markdown
# [项目名称] 项目状态跟踪
## 📋 项目概览
## 📊 项目总体进度  
## 📋 当前Sprint任务状态
## 📝 Sprint开发日志
## 📊 项目度量指标
## 🚨 风险和问题跟踪
## 📝 文档更新信息
```

### 2. 章节内容要求

#### 2.1 项目概览 📋

**必需字段**:
```markdown
> **项目名称**: [项目名] - [简短描述]
> **项目愿景**: [一句话愿景描述]
> **项目开始时间**: YYYY-MM-DD
> **当前状态**: [当前Sprint] [优先级状态] 🚨
> **项目阶段**: [开发阶段] ([完成度]%完成度)
```

**必需子章节**:
- 🎯 项目核心特性 (按技术领域分类)
- 📊 项目价值与成果 (技术突破/用户体验/开发效率)
- 🔄 最近完成的重大突破

#### 2.2 项目总体进度 📊

**必需子章节**:
- **整体开发进度概况** 🎯
- **六个Sprint概览** (表格形式)
- **当前Sprint状态**
- **🔄 接下来的项目规划**

**进度概况格式**:
```markdown
**项目总体完成度**: **XX%** (状态描述)
**开发阶段状态**: 
- ✅/🔥/⏳ **阶段名称** (XX%) - 阶段描述
**累计工时投入**: **XX小时** / 计划**XX小时** (XX.X%)
**关键里程碑达成**: 
- 🏆 **里程碑名称**: 描述 (日期)
```

#### 2.3 当前Sprint任务状态 📋

**必需内容**:
- Sprint概览 (目标/时间/状态/进度)
- 关键问题清单 (问题识别和解决状态)
- 任务进度详情 (按优先级P0/P1分组)
- 最近完成的任务 (详细成果展示)

**任务格式**:
```markdown
- [X/空格] **任务X.X**: 任务名称 (Xh) ✅/🎯 **状态** (日期)
  - ✅ 具体完成项1
  - ✅ 具体完成项2
```

#### 2.4 Sprint开发日志 📝

**必需格式**:
```markdown
### YYYY年MM月DD日 - 事件描述 ⏰

**阶段**: 具体工作阶段描述

#### 第X阶段：阶段名称 (时间范围, Xh)

**时间-时间** 🔍 **具体工作内容**
- 详细工作步骤1
- 详细工作步骤2
```

#### 2.5 项目度量指标 📊

**必需指标类别**:
- 验收通过情况 (通过率/完成度/质量评估)
- 开发效率 (任务完成/时间控制/代码质量)

#### 2.6 风险和问题跟踪 🚨

**必需分类**:
- 当前风险 (未解决的风险项)
- 已解决问题 (历史问题记录)
- 待解决问题 (已识别待处理)

#### 2.7 文档更新信息 📝

**必需字段**:
```markdown
- **最后更新时间**: YYYY年MM月DD日 HH:MM
- **更新内容**: 主要更新内容
- **更新人员**: 更新人姓名
- **文档版本**: vX.X
- **重大里程碑**: 当前重要成就
```

## 🎯 内容编写规范

### 1. 语言风格要求

#### 1.1 专业性原则
- 使用准确的技术术语
- 提供具体的数据和指标
- 避免模糊的描述词汇

#### 1.2 简洁性原则
- 每个描述控制在1-2句话内
- 使用bullet points清晰列举
- 避免冗长的段落描述

#### 1.3 一致性原则
- 统一的时间格式: YYYY-MM-DD
- 统一的状态图标: ✅已完成 🔥进行中 ⏳待开始 ⏸️暂停
- 统一的优先级标记: 🔥P0关键级 🎯P1重要级

### 2. 数据填写规范

#### 2.1 进度数据
- 百分比精确到整数，特殊情况可到小数点后1位
- 工时统计精确到小时，使用整数
- 完成时间使用具体日期，避免相对时间

#### 2.2 状态标识
- 完成状态: ✅ (绿色对勾)
- 进行中: 🔥 (火焰，表示活跃)
- 待开始: ⏳ (沙漏，表示等待)
- 暂停: ⏸️ (暂停按钮)
- 取消: ❌ (红叉，慎用)

#### 2.3 优先级标记
- P0关键级: 🔥 (必须立即处理)
- P1重要级: 🎯 (需要及时处理)
- P2一般级: 📋 (常规处理)

### 3. 技术描述规范

#### 3.1 技术成果描述
- 使用量化指标 (性能提升X倍，时间从X秒到Y秒)
- 描述具体技术方案 (算法名称，架构模式)
- 说明业务价值 (解决了什么问题，带来什么收益)

#### 3.2 问题描述规范
- 问题现象描述要具体
- 根因分析要深入
- 解决方案要完整
- 验证结果要明确

## 📊 质量保证规范

### 1. 更新频率要求

#### 1.1 实时更新项
- 当前Sprint状态 (每日更新)
- 任务完成状态 (任务完成时立即更新)
- 风险问题状态 (发现/解决时立即更新)

#### 1.2 定期更新项
- 项目总体进度 (每周更新)
- 度量指标 (每Sprint更新)
- 开发日志 (每个工作日更新)

### 2. 数据一致性检查

#### 2.1 数据关联性检查
- Sprint表格中的工时数据应与总体进度一致
- 任务完成百分比应与实际任务数量匹配
- 度量指标应与实际任务记录一致

#### 2.2 时间逻辑检查
- 开始时间不能晚于结束时间
- 当前日期不能早于项目开始时间
- Sprint时间不能重叠

### 3. 内容完整性检查

#### 3.1 必需信息检查清单
- [ ] 项目基本信息完整
- [ ] 所有Sprint都有状态记录
- [ ] 当前任务有详细进度
- [ ] 风险问题有跟踪记录
- [ ] 文档更新信息准确

#### 3.2 格式规范检查清单
- [ ] 章节标题使用正确的emoji
- [ ] 表格格式完整规范
- [ ] 链接地址有效可访问
- [ ] 代码块使用正确语法高亮

## 🔄 维护工作流程

### 1. 日常维护流程

#### 1.1 任务完成时
1. 更新任务状态为 ✅
2. 记录完成时间
3. 更新Sprint进度百分比
4. 添加到"最近完成任务"
5. 更新开发日志

#### 1.2 新问题发现时
1. 添加到"当前风险"或"待解决问题"
2. 评估优先级和影响范围
3. 制定解决计划
4. 分配责任人

#### 1.3 问题解决时
1. 从待解决移动到已解决
2. 记录解决方案和完成时间
3. 更新相关风险状态
4. 总结经验教训

### 2. 定期评审流程

#### 2.1 每周评审 (周五)
- 检查数据一致性
- 更新项目总体进度
- 评估风险和问题状态
- 计划下周工作重点

#### 2.2 每Sprint评审
- 归档完成的Sprint
- 更新度量指标
- 评估Sprint目标达成情况
- 规划下个Sprint

## 🚨 常见问题和解决方案

### 1. 数据不一致问题

**问题**: Sprint表格工时与总体进度不匹配
**解决**: 建立数据源统一原则，以Sprint详细记录为准

**问题**: 任务完成百分比计算错误
**解决**: 使用公式 完成任务数/总任务数×100%，四舍五入到整数

### 2. 更新不及时问题

**问题**: 状态更新滞后导致信息过时
**解决**: 建立更新提醒机制，关键节点强制更新

### 3. 描述质量问题

**问题**: 技术描述过于简单或过于复杂
**解决**: 遵循"一个核心观点+三个支撑要点"原则
